Exchanger简介

Exchanger是一个多线程协作类，用于进行线程间的数据交换。它提供一个同步点，在这个同步点，两个线程可以交换彼此的数据。两个线程通过exchange方法交换数据，那个线程先执行exchange方法，会一直等待另一个线程执行exchange方法，当两个同时到达同步点，两个线程可以交换数据，将本线程生产的数据传输给对方。

线程池的好处

  • 降低资源消耗，通过重复利用已创建的线程降低线程创建和销毁造成的损耗
  • 提高相应速度，当线程到达时候任务可以不需要等待线程创建就能立即执行
  • 提高线程的可管理性，统一分配、调优、监控

线程池实现原理

当向线程池提交一个任务，线程池的处理方式如下图所示

  • 如果线程池中的线程小于corePoolSize时就会创建新线程直接执行任务
  • 如果线程池中的线程大于corePoolSize就会暂时把任务存放在工作队列workQueue中等待执行
  • 如果工作队列workQueue也满时，当线程数小于线程池数maximumPoolSize时就会创建新线程来处理，而线程数大于最大线程池数maximumPoolSize时就会执行饱和策略。

乐观锁和悲观锁

发表于 2017-12-12 |

乐观锁概念

维基百科定义：
在关系数据库管理系统里，乐观并发控制（又名“乐观锁”，Optimistic Concurrency Control，缩写“OCC”）是一种并发控制的方法。它假设多用户并发的事务在处理时不会彼此互相影响，各事务能够在不产生锁的情况下处理各自影响的那部分数据。在提交数据更新之前，每个事务会先检查在该事务读取数据后，有没有其他事务又修改了该数据。如果其他事务有更新的话，正在提交的事务会进行回滚。乐观事务控制最早是由孔祥重（H.T.Kung）教授提出。

悲观锁概念

在关系数据库管理系统里，悲观并发控制（又名“悲观锁”，Pessimistic Concurrency Control，缩写“PCC”）是一种并发控制的方法。它可以阻止一个事务以影响其他用户的方式来修改数据。如果一个事务执行的操作读某行数据应用了锁，那只有当这个事务把锁释放，其他事务才能够执行与该锁冲突的操作。

高并发业务场景

在高并发业务场景中，大量请求同时发起，会导致库存数据不正确。可以用乐观锁和悲观锁来解决。

这里使用select for update的方式利用数据库开启了悲观锁，锁定了id=1的这条数据。之后使用commit提交事务并释放锁，这样下一个线程过来拿到的就是正确的数据。

悲观锁一般是用于并发不是很高，并且不允许脏读等情况。但是对数据库资源消耗较大。

使用乐观锁追求更好的性能，支持更高并发

乐观锁是首先假设数据冲突很少，只有在数据提交修改的时候才进行校验，如果冲突了则不会进行更新。通常的实现方式增加一个version字段，为每一条数据加上版本。每次更新的时候version+1，并且更新时候带上版本号

乐观锁在实际应用相对较多，它可以提供更好的并发访问，并且数据库开销较少，但是有可能存在脏读的情况。

多线程临界区的概念

  • 多线程访问访问交替访问同一个对象，这个对象被称为临界资源。
  • 一次只有一个线程使用临界资源，当一个线程获得临界资源使用权限，其他线程必须等待。
  • 一个线程获取临界资源不能无限制的使用下去，必须在有限时间内释放。
  • 线程不安全，多线程访问同一个对象，可能会造成数据不一致。

内存可见性

多线程情况下，一个线程对内存共享变量做出了修改，没有及时改变共享变量的值。另一个线程也要对共享变量进行修改，此时拿到的变量还没有被修改，也就是一个线程对内存变量的修改，另一个线程不能马上看到，甚至永远也看不到，发生了数据错误，这个就是内存可见性。

加锁可以解决这个问题。java中常用的枷锁方式有：

  • 使用synchronized关键字
  • 使用显示锁 lock
  • 使用volatile

synchronized

  • 同步代码块
    synchronized(o){ } 对o加锁的同步代码块，只有拿到锁标记的线程才能够进入对o加锁的同步代码块。
  • 同步方法
    synchronized作为方法修饰符修饰的方法被称为同步方法，整个方法都是一个代码块。

ReentrantLock

ReentrantLock与synchronized作用类似,更加灵活。

java-多线程生命周期

发表于 2017-12-10 | 分类于 java |

线程的5种状态

1.新建状态（New） 新创建了一个线程对象。
2.就绪状态（Runnable） 线程对象创建后，其他线程调用了该对象的start()方法。该状态的线程位于可运行线程池中，变得可运行，等待获取CPU的使用权。
3.运行状态（Running） 就绪状态的线程获取了CPU，执行程序代码。
4.阻塞状态（Blocked） 阻塞状态是线程因为某种原因放弃CPU使用权，暂时停止运行。直到线程进入就绪状态，才有机会转到运行状态。阻塞的情况分三种：

  • 等待阻塞：运行的线程执行wait()方法，JVM会把该线程放入等待池中。
  • 同步阻塞：运行的线程在获取对象的同步锁时，若该同步锁被别的线程占用，则JVM会把该线程放入锁池中。
  • 其他阻塞：运行的线程执行sleep()或join()方法，或者发出了I/O请求时，JVM会把该线程置为阻塞状态。当sleep()状态超时、join()等待线程终止或者超时、或者I/O处理完毕时，线程重新转入就绪状态。
    5.死亡状态（Dead）：线程执行完了或者因异常退出了run()方法，该线程结束生命周期。
